# 🗓️ PRD: NANA PLANNER

## 1. プロジェクトの背景とビジョン (Background & Vision)
既存の「1週間単位のタイムブロッキングツール（Obsidian Weekplan）」の反省から生まれた、より解像度の高い1日単位のタスク管理アプリです。
[cite_start]1週間の大きな石を置く前に、まずは「1日の石」を確実に配置し、実行できる必要があります [cite: 1]。

本ツールの目的は、**「現在の点（Now）」から「次の点（絶対に動かせないマイルストーン）」までの繋がりと残り時間を可視化し、リアルタイムな焦燥感と見通しを提供すること**です。

## 2. ターゲット環境と技術スタック (Environment & Tech Stack)
本プロジェクトは、以下の制約と環境下で動作・開発されることを前提とします。

* **開発環境**: macOS (開発・テスト・Vibe Coding用)
* **本番環境**: Windows 10/11 (会社のPC)
* **最大の制約**: 会社のWindows PCは**管理者権限がない（申請制）**ため、インストーラを経由せず、ユーザー領域で完結する「ポータブルな `.exe`（単一ファイル）」である必要があります。

### 採用技術
* [cite_start]**バックエンド/ガワ**: **Tauri v2** (超軽量・省メモリなWindows常駐アプリを実現するため [cite: 2])
* **フロントエンド**: **React + TypeScript + Vite + Tailwind CSS**
* **UIインタラクション**: **dnd-kit** (ドラッグ＆ドロップ実装用)
* **CI/CD (ビルド)**: **GitHub Actions** (Mac開発環境からWindows用のポータブル `.exe` を自動生成するため)
* **データストア**: ローカルの Obsidian Vault 内の YAML/JSON ファイルを直接読み書き。
* [cite_start]**外部連携**: Pythonスクリプト (`pywin32` 経由) をTauriから呼び出し、ローカルのクラシックOutlookから予定を取得 [cite: 3, 4]。

## 3. コアとなるUX/UI要件 (Core UX/UI Requirements)
既存のカレンダーライブラリ（FullCalendar等）は使用せず、ReactとTailwindによる**「自前実装の縦型タイムライン」**を構築します。

### A. 非稼働時間のワープ（アコーディオン）UI
* 1日の終わりでタイムラインを分断せず、今日と明日を地続きで表示します。
* [cite_start]例：今日の業務終了（19:00）から翌日の始業（09:00）までの時間は、大きな空白にせず、**細い帯状のアコーディオンUIとして折りたたんで（ワープさせて）表示**します [cite: 5]。
* これにより、「今日の18時に終わらなければ、猶予は明日の9時〜10時の1時間しかない」という現実を直感的に見せます。

### B. マイルストーンと「重力」の可視化
* [cite_start]Outlookから同期された「会議」などの固定予定は、絶対に動かせない**マイルストーン（執着点）**として配置されます [cite: 6, 9]。
* ユーザーは、そのマイルストーンに向けて必要なToDo（弾）をタイムラインの空き時間にドラッグ＆ドロップで配置します。

### C. タイムプログレスと焦燥感のアラート
* 「現在時刻」を示す横線（Now Line）が上から下へ移動します。
* [cite_start]タスクの予定完了時刻を「現在時刻の線」が追い越した（終わっていない）場合、**そのタスクのブロック全体が赤く点滅（またはハイライト）**し、視覚的なアラートを出します [cite: 7]。
* [cite_start]次のマイルストーンまでの「実質的な残り時間（カウントダウン）」を動的に表示します [cite: 8, 9]。

## 4. コーディングエージェントへの指示 (Instructions for Vibe Coding)
AIエージェント（あなた）は、以下のルールに従ってコードを出力してください。

1.  **FullCalendarの禁止**: タイムラインは `div` と Tailwind の Grid/Flexbox を駆使して自前で計算・描画してください。時間をピクセル（高さ）に変換するロジックをシンプルに保つこと。
2.  **Tauriのポータブルビルド**: GitHub Actionsのワークフロー（`.github/workflows/build.yml`）を作成する際は、インストーラ（msi/nsis）ではなく、`target/release/` に出力される単一の `.exe` ファイルを Artifact としてアップロードするように記述してください。
3.  **Rustコードの最小化**: Tauri側のRustコードは「ファイルの読み書き」と「Pythonコマンドの実行」のみに留め、複雑なロジックはすべてフロントエンド（React/TypeScript）で処理してください。
4.  **UIモックアップ先行**: 最初からバックエンドを繋ぎこまず、まずはReact上で「ダミーデータを用いたタイムラインUI（時間ワープ、赤色アラート含む）」のコンポーネントを完成させることを優先してください。
